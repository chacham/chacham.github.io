# 정적 사이트 생성기(Static Site Generator)?

정적 사이트는 정적으로(static하게) 만들어져있는 상태로 전달되는 웹 페이지(site)를 말한다. 그러니 정적 사이트 생성기는 그런 정적 사이트를 만들어주는 생성기이다. 적고보니 이해하는데 전혀 도움이 안되는 문장.. (사실 만들어져있는 상태로 전달되는 것은 좁은 말이다. 뒤에서 다시 이야기)


## 정적 웹 사이트(Static Web Site)

초창기의 웹 페이지들은 대부분 정적인 사이트였다고 한다. (어디서 주워들은 글이라 어느정도까지 초창기를 말하는건진 잘 모르겠다.) HTTP를 통해 정적인 파일들을 주고받으며, 하이퍼링크로 다른 웹페이지로 떠돌아다니기. Progress bar가 차오르는걸 파도라고 생각하며 웹 surfing을 하던 때.

내가 어릴 때에는(인터넷의 초창기라고는 절대 할 수 없는 시절) 그래도 정적 사이트가 꽤 많았던 기억이 난다. 나모 웹 에디터로 내 홈페이지를 만드는 책들이 꽤 많았고, 나만의 공간에 "~~의 게임나라" 같은 개인 홈페이지가 많았다! 그리고 동적인 사이트도 그렇게 동적이지는 않았다. 기껏해야 글을 쓰면 목록이 바뀌는 게시판정도. (사실 과하게 동적인? 페이지도 많았다. 마퀴태그로 현란하게 꾸며놓은 곳과 무한으로뜨는 팝업창으로 날 괴롭이던 낚시글들...)


## 동적 웹 사이트(Dynamic Web Site)

정적인 사이트 얘기만 하고있었는데, 사실 요즘은 완전 정적인 사이트는 찾아보기 힘들어진 것 같다. 구글에서 몇페이지 넘어가면 있는 아주 낡은 글들(잃어버린 일기장같은 느낌이 나는)이라거나, 지금 여기같은 정적 사이트 생성기로 만든 블로그처럼 정보 전달용 페이지들정도 말고는 최근에 본 기억이 없다.

동적 웹 사이트란 완성된 상태로 어디 있는걸 내어주는게 아니라, 웹사이트를 보여달라고 할 때마다 재료를 이용해 어디선가 웹사이트를 완성해서 보여주는것을 말한다. 그런데 그 요리(웹사이트 생성)를 주방장(서버)이 하는지, 내(브라우저)가 하는지에 따라서도 조금 다르다.

### 서버 사이드 동적 웹 페이지(Server-side Dynamic Web Page)

먼저 서버에서 요리를 완성해서 나에게 보내주는 서버 사이드 동적 웹 사이트가 있다. 성능상의 이유와(게시판에 글이 많이많이 있는데, 이걸 한번에 내려주면 서로서로 힘드니 가볍게 필요한것만 채워넣어 내려주기) 보안상의 이유로(내가 볼 수 없는것도 모두 알려주고 알아서 보라고 하면 불안하니까) 서버는 클라이언트(크롬, 사파리 등 웹 브라우저)가 요청하면 필요한 데이터만 가져와서 웹페이지를 완성시켜 내려줄 수 있다.

로그인 여부, 글이나 댓글 목록의 변화, 현재 시간 등등 어떤 이유로든 다른 웹사이트를 서버에서 내려줄 수 있다. 그리고 서버에서 웹사이트를 완성해서 보내주면, 클라이언트에서는 이게 동적인 사이트인지 정적인 사이트인지 굳이 구분할 필요도, 구분할 방법도 없다.(주방장이 쿠키를 사와서 주든, 구워서 주든 어쨌든 바로 먹을 수 있으니!) 이렇게 서버에서 웹페이지를 완성해서 클라이언트에 내려주는것을 서버 사이드 렌더링이라고 한다.

스프링을 처음 배울 때 서블릿과 JPA를 공부했었고, PHP를 찍어먹어봤고, Node를 사용하면서는 pug와 ejs를 써봤던 것 같다. 이것들을 서버사이드에서 웹페이지를 만들어서 내려줄 때 썼다. 서버 사이드 렌더링 개념을 잘 모를땐, 브라우저에서 php의 원본 코드를 알아낼 수 있는 방법이 없을지 한참 알아보며 힘들어했던 기억이 난다. (php 코드가 html 안에 들어가있는 모양새라서, 서버사이드에서 완성해서 내려준단걸 잘 이해하지 못했던 듯) 보안상 헛점이 있는 서버가 아니라면 php 원본 코드를 볼 수 없단걸 알게된것은 생각보다 꽤 많은 시간이 지난 뒤였다.
